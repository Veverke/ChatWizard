# Search Bar Fix — Full Post-Mortem

## The Problem

ChatWizard's Session Viewer has an in-webview search bar (type to highlight, Enter/▲▼ to navigate). It was never functional. Additionally, after early fix attempts, **clicking a chat session in the tree stopped loading any content at all** — a regression that made the entire panel blank.

The panel is a VS Code `WebviewPanel`. Content is set by assigning to `panel.webview.html`. VS Code internally routes that HTML through `document.write()` in an iframe. Any JavaScript `SyntaxError` inside that HTML aborts the `document.write()` call, which prevents VS Code from injecting `acquireVsCodeApi()` — meaning the extension↔webview message channel is never established and nothing can load.

---

## Chronological Attempts

### 1. Direct inline search JS in `getHtml()` — Did not fix it

The original panel generated a complete HTML page (including session content + search JS) on every `show()` call and assigned it to `panel.webview.html`. The search bar existed in the HTML but never responded. Root cause was not yet known.

---

### 2. Suspected CSP blocking scripts — Partially correct diagnosis, wrong fix

Hypothesis: the `Content-Security-Policy` header with `script-src 'nonce-${nonce}'` was blocking the inline search script.

Observation: VS Code's `toContentHtml()` round-trips the HTML through `DOMParser → outerHTML`. Chrome strips `nonce` attribute *values* from `outerHTML` serialization (a security feature to prevent CSS-based nonce exfiltration). So `<script nonce="abc">` becomes `<script nonce="">` in the output, which the CSP then blocks.

Fix attempted: switch to `script-src 'unsafe-inline'`.

Result: this was the **correct CSP fix** and necessary, but sessions still didn't load — there was a deeper problem.

---

### 3. Suspected stale persisted HTML cache — Correct diagnosis, temporary workaround

VS Code persists webview HTML per origin (keyed on `viewType` → UUID). Once a broken HTML document is written for an origin, VS Code keeps serving it from cache on every reload, even after the source code is fixed.

Fix attempted: rename the viewType from `chatwizardSession` → `chatwizardSession2` to force a new UUID with no cached HTML.

Result: new origin `d8f163fa` was created — but it *also* failed with the same `SyntaxError`. The HTML itself was still broken, not just the cache.

> **Side note on viewType renames:** each rename leaves an orphaned persisted origin. Final viewType is `chatwizardSession3`. If the panel ever breaks and produces a blank screen with no obvious code change, try renaming the viewType again to clear any newly-cached broken HTML.

---

### 4. Shell + postMessage rewrite — Correct architectural fix, but escape bug survived

Hypothesis: the real problem is that session content (user messages, code blocks, etc.) flows through `document.write()`. Any special character in a user message could produce a SyntaxError.

Fix: rewrite `sessionWebviewPanel.ts` to use the **shell + postMessage** pattern already used by other panels (analytics, codeblocks, timeline):

- `panel.webview.html` is set **once** with a static, content-free shell (no user data).
- Session content is sent via `panel.webview.postMessage({ type: 'render', ... })`.
- The shell's `window.addEventListener('message')` renders content via `innerHTML`.

This is architecturally correct and permanently solves the `document.write()` / user-content problem. Renamed viewType to `chatwizardSession3` (new UUID `f17bb9b9`) to clear the cache again.

Result: sessions **still** didn't load. The shell itself — containing no user content — was still producing a `SyntaxError`. A new root cause was present inside the shell's own static JavaScript.

---

### 5. Template literal escape sequences — THE FIX ✅

The shell HTML is generated by a TypeScript template literal (backtick string). Inside the shell, a JavaScript syntax-tokenizer function contained these lines:

```typescript
// TypeScript template literal — BROKEN:
var nl = code.indexOf('\n', i);            // line 397
var nh = code.indexOf('\n', i);            // line 401
if (code[j] === '\\') { j+=2; continue; } // line 406
```

**What TypeScript does with these:**

| TypeScript source | Characters in output JS |
|---|---|
| `'\n'` inside template literal | literal newline (char code 10) |
| `'\\'` inside template literal | single backslash `\` |

**What the output JS looks like:**

```javascript
// After template expansion — INVALID JS:
var nl = code.indexOf('
', i);          // newline inside single-quoted string → SyntaxError
if (code[j] === '\') { ... }             // backslash escapes closing quote → SyntaxError
```

A bare newline character inside a JS single-quoted string literal is an unterminated string literal — a hard `SyntaxError`. Same for a lone `\` before the closing quote. This caused `document.write()` to throw even for the fully static shell with zero user content.

**The fix** — match the pattern already correctly used in `codeBlocksPanel.ts`:

```typescript
// TypeScript template literal — CORRECT:
var nl = code.indexOf('\\n', i);             // → '\n' in output (valid escape)
var nh = code.indexOf('\\n', i);             // → '\n' in output
if (code[j] === '\\\\') { j+=2; continue; } // → '\\' in output (valid backslash check)
```

This was the one change that fixed everything: sessions loaded, `acquireVsCodeApi()` was injected, the `ready` handshake fired, content rendered, and the search bar worked.

---

## Why It Was So Hard to Find

**1. The error message is misleading.**
`SyntaxError: Failed to execute 'write' on 'Document': Invalid or unexpected token` points at VS Code's own `index.html:1086` — not at the offending code in `sessionWebviewPanel.ts`. There is no stack frame pointing at the real source.

**2. The cache multiplied iterations.**
Every broken version got cached per-origin. Each fix attempt required renaming the viewType to get a fresh origin, adding a red-herring step that obscured whether a fix actually worked.

**3. The nonce bug and the escape bug produced identical symptoms.**
Both cause `document.write()` to fail. Both kill `acquireVsCodeApi()`. Both result in a blank panel. Fixing the nonce issue (step 2) was necessary but not sufficient, so the escape bug stayed hidden behind it.

**4. The escape bug only exists at the TypeScript→JS boundary.**
The JavaScript code *looks* correct when read in isolation — `'\n'` and `'\\'` are valid JS. The bug only manifests because the code lives inside a TypeScript template literal, where `\n` and `\\` are already interpreted before the string is emitted into the generated HTML.

---

## Key Rule for Future Reference

> **Inside a TypeScript template literal that generates JavaScript source code, every escape sequence must be doubled.**
>
> | Desired output JS | Write in TypeScript template |
> |---|---|
> | `'\n'` | `'\\n'` |
> | `'\\'` | `'\\\\'` |
> | `'\t'` | `'\\t'` |
>
> Anything else will be interpreted by TypeScript first, potentially producing literal control characters that break the generated JS syntax.
>
> **Canonical correct example in this codebase:** `src/codeblocks/codeBlocksPanel.ts`, tokenizer section.

## VS Code Webview Mechanics (Reference)

These internals explain why the symptoms were so confusing:

- `panel.webview.html = html` → VS Code calls `toContentHtml(html)` which parses via `DOMParser`, injects `_vscodeApiScript` (the `acquireVsCodeApi` implementation), serializes back via `outerHTML`, then calls `contentDocument.write(result)` at `index.html:1086`.
- If `document.write()` throws, `acquireVsCodeApi` is never defined in the webview's window. Any call to it in the webview JS throws `ReferenceError`, meaning the `ready` message is never sent, and the extension never sends the `render` payload — blank panel.
- Chrome strips `nonce` attribute values during `outerHTML` serialization. Never use `script-src 'nonce-...'` in a VS Code webview CSP. Use `script-src 'unsafe-inline'` instead (matching analytics, codeblocks, timeline, promptLibrary panels in this project).
- VS Code caches webview HTML per origin (viewType → UUID). A broken document gets cached and re-served on every reload. To force a clean slate: change the `viewType` string in both `WebviewPanel.createOrShow()` and `registerWebviewPanelSerializer()`.
